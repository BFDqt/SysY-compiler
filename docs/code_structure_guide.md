## 1. 核心目录与文件结构一览

可以把这个结构当作地图展示给老师，体现出项目极高的**模块化解耦**程度。

```text
├── src/main.cpp                        # 【1】总场控：负责串联所有模块的入口函数
├── src/frontend/                       # 【2】大前端：字符串源码 -> 抽象语法树(AST)
│   ├── compiler2022_x/FrontEnd/YaccLex/src/sysy.l # Flex 词法规则文件：负责分词
│   ├── compiler2022_x/FrontEnd/YaccLex/src/sysy.y # Bison 语法规则文件：负责组装树
│   └── driver.cpp                      # 对象树适配器：将前端临时树转化为核心 AST
├── include/sysy/ast.hpp                # 【3】数据桥梁：AST 核心数据结构设计
├── src/semantic/SemanticAnalyzer.cpp   # 【4】编译器大脑：一趟遍历，完成检查并生成 Koopa IR 
└── src/backend/RiscV.cpp               # 【5】打工人：纯粹的 Koopa IR -> RISC-V 汇编翻译机器
```

## 2. 核心文件与关键函数指北

答辩展示代码时，直接告诉老师：“我的 XXX 逻辑实在 XXX 文件的 XXX 函数里控制的。”

### 【主入口】 `src/main.cpp`
整个项目的中央调度室。
- **`main()` 函数**：
  1. 通过 `parseArgs` 处理入参。
  2. 调用 `Driver::parseFile(...)` 获取 `AST`。
  3. 调用 `SemanticAnalyzer::compile(...)` 把 AST 变成 `Koopa IR`。
  4. 如果指定是 `-riscv` 模式，将上一步返回的 IR 交给 `RiscVGenerator::generate` 进行目标代码生成。

### 【AST 核心定义】 `include/sysy/ast.hpp` 和 `src/ast/ast.cpp`
前端与语义层的严格分界线。
- **核心类设计**：`Program`(顶层组件), `TopLevel`, `FuncDef`(函数定义), `Stmt`(语句), `Expr`(表达式)。
- **设计选择**：没有搞深度的面向对象多态（继承继承再继承），而是采用了 **类型枚举（如 `ExprKind`, `StmtKind`） + 联合属性** 的平铺设计。好处是极大简化了后续利用 `switch-case` 遍历树的开发体验。

### 【语义大脑与 IR 生成】 `src/semantic/SemanticAnalyzer.cpp`
全项目最长的文件 (1500+ 行)，是最核心的逻辑中枢。它的任务是：顺着 AST 检查类型对不对、作用域对不对，查完顺手把合法的 Koopa 中间代码吐出来。
- **`Impl::compile()`**：主入口。在这里首先调用了内置函数注册，随后扫描所有的顶层结构。
- **`collectFunctionSignatures()`**：**（重点）** 函数预扫描。它会先把全局函数的参数和返回值收集进符号表。这是为了解决**函数在定义之前就被互相调用**以及**递归**的问题。
- **`compileFunction() / compileStmt() / compileExpr()`**：标准的 AST 树形向下递归遍历入口。遇到不同的阶段就执行不同的检查逻辑。
- **`evalConstExpr()` 族**：处理所谓的“编译期常量”。比如数组的维度、或者 `const int a = 2 * 3;` 这种在编译期必须算出结果的东西。
- **`fillConstInitializer() / fillRuntimeInitializer()`**：**（重点）** 这是用来解决 SysY 里最让人头疼的多维数组初始化问题的函数。它确保像 `{{1, 2}, 3}` 这种嵌套的大括号能被精确展开，并在未填满的地方**补 0**。

### 【目标代码后端】 `src/backend/RiscV.cpp`
非常克制、纯粹的后端实现。它不知道“AST”，它只认识上一步产出的 Koopa 中间代码。
- **`RiscVGenerator::generate()`**：入口函数，通过调用第三方库解析 Koopa 文本为 `raw program` 内存结构后，开始真正遍历。
- **`RawEmitter::buildFrame()`**：**（重点）** **栈帧计算器**。计算当前这个函数如果要转成汇编，需要在内存栈上预留多大的空间（存参数需要多少、局部变量需要多少、以及有没有多余的参数溢出块）。
- **`RawEmitter::emitInstruction()`**：一个巨型的指令翻译机。把 Koopa 中的 `load`、`store`、`call` 等指令一对一对应转化为对应的 RISC-V 操作及栈指针移动。

## 3. 项目可拿来背书的 5 个技术亮点

答辩的目的是展现你的设计思考（Trade-off）。如果老师问“你项目有什么出彩点或难点”，直接抛出以下 5 点：

### 亮点一：前端和后端的高解耦性
很多萌新编译器喜欢把字符串一路处理到汇编混写在一起。本项目的亮点是通过引入 **自定义 AST** 和 **Koopa IR 中间态** 实现严格分离。
> **话术**：我做了极强的模块隔离。后端的 `RiscV.cpp` 不需要知道 SysY 的任何语法规则，只要你喂给它的 Koopa 是符合语法的，它就能闭着眼睛出 RISC-V 汇编。这极大降低了重构和 Debug 的复杂度。

### 亮点二：短路求值的图模型转换
逻辑与 `&&` 和逻辑或 `||` 不是算术运算，如果左边决定了结果，右边不能执行。
> **话术**：我没有把逻辑操作当成简单的二元运算符在同一条语句里生成，而是直接在生成 Koopa 时**强制阻断**，把它们转为了**条件分支流（Basic Blocks）**。先申请一个临时结果槽（内存），在通过 `branch` 跳跃赋值，从而从底层汇编上实现了精确的“短路不发生副作用”。

### 亮点三：数组退化与多维数组传参的指针识别
C语系的数组是一大痛点，形参和实参看起来是数组，其实是指针。
> **话术**：我在语义层的符号表使用了 `isArrayParam` 专门标记这种情况。在 `SemanticAnalyzer.cpp` 处理参数时，若是真数组作为实参，采用 `getelemptr base, 0` 实现首元素退化传递；由于将这两种情况分类处理好了，即使是 `a[][3]` 的调用也能完美通过类型长度判定和汇编寻址生成。

### 亮点四：优雅的 Float 计算降维逃课法
Koopa 原生比较适合跑 integer 处理，要自己加入原生 float 非常费神。
> **话术**：系统语义中，我把所有 float 常量按 IEEE 754 转换为 32 位整型的**位模式 (bit pattern)** 保留在底层运算槽中。加减乘除和显性转换，我设计成了注入式的内部 Helper 函数调用 (`__sysy_fadd` 等)，最后仅在发现调用过这些 helper 时，由后端追加纯粹的 RV32F 汇编片段。这样做让主体 IR 和后端框架极其清爽。

### 亮点五：安全且易于调试的「全部落栈」策略
后端没有去弄繁重的图染色寄存器分配算法。
> **话术**：为了追求 100% 的内存安全与调用约定（Calling Convention）合法，后端选用了最保守但绝对正确的做法——所有的参数、临时变量统统在 `buildFrame` 里分配栈内存，执行任何运算时 `Load` 到临时寄存器，算完直接 `Store` 返还栈中。这就避免了寄存器覆盖等幽灵 Bug。
