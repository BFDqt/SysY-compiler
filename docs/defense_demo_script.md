# SysY 答辩现场演示与防翻车预案

这份文档是为你“站在讲台上”和“面临老师刁钻提问”时准备的**最后一道防线**。它包含了可以直接复制粘贴的现场演示流程，以及在被问到项目缺陷时如何高情商圆场的“护身符”。

---

## 1. 现场演示剧本 (直接照着做)

在答辩现场，不要临时现找代码测。直接使用以下流程，向老师证明你的编译器能走通全链路。

### 第一步：展示源码
准备一个容易解释的样例，比如 `tests/positive/02_expr.sy` 或 `04_control.sy`。
> **话术**：老师好，我先用一个基础的表达式/控制流代码来演示编译器的全链路。这几行代码测试了程序的 XXX 逻辑。

### 第二步：编译你的编译器
> **话术**：首先，我在 Docker 环境中用 CMake 构建编译器自身。
```bash
# 在终端中敲入
cmake -S . -B build
cmake --build build -j
```

### 第三步：生成 Koopa IR (中间代码)
> **话术**：接着，我把这段 .sy 代码编译成中间表示 Koopa IR，请看生成的代码。
```bash
./build/compiler -koopa tests/positive/04_control.sy -o /tmp/demo.koopa
cat /tmp/demo.koopa
```
**(指着屏幕重点指出)**：“老师您看，这里的 `alloc` 代表我分配了局部栈内存，每一条 `load` 和 `store` 都体现了严格的值传递。”

### 第四步：生成 RISC-V 汇编
> **话术**：然后，我的后端将这套 IR 翻译为 RISC-V 目标机器汇编。
```bash
./build/compiler -riscv tests/positive/04_control.sy -o /tmp/demo.S
cat /tmp/demo.S
```
**(指着屏幕重点指出)**：“您看这里开头的 `addi sp, sp, -XX`，这是我后端在为这个函数建立准确大小的栈帧。”

### 第五步：链接跑通，展示结果
> **话术**：最后我们通过 clang 和 qemu 模拟运行它，验证返回值完全正确。
```bash
clang /tmp/demo.S -c -o /tmp/demo.o -target riscv32-unknown-linux-elf -march=rv32imf -mabi=ilp32
ld.lld /tmp/demo.o -L"/opt/riscv/lib" -lsysy -o /tmp/demo
qemu-riscv32-static /tmp/demo
echo $?
```
**(指着屏幕)**：“这里输出的数值，和 SysY 源码中 return 期望的值是完全吻合的。演示完毕。”

## 2. 老师指着生成的代码问，怎么圆？

如果老师随便指着你终端里生成的乱码问“这句是什么意思”，记好以下几个高频词：

### Koopa IR 高频词汇
*   `@xxx = alloc i32`: 申请了一个 4 字节的局部变量，类似 C 里的 `int x;`。
*   `store %1, @xxx`: 把变量存进刚才申请的内存里。
*   `%2 = load @xxx`: 从内存里把值读出来，放到临时变量 `%2` 里准备运算。
*   `br %1, %then, %else`: 分支跳转，这是 `if` 或者 `while` 或者**短路求值**的核心跳跃点。
*   `getptr` / `getelemptr`: 老师问这是啥，答：“这是处理数组和指针退化时的偏移量计算指令”。

### RISC-V 汇编高频词汇
*   `sw a0, 12(sp)` / `lw t0, 12(sp)`：如果出现大量的这种 `sw` 和 `lw`，老师问为什么这么啰嗦。答：“**这是因为我采取了保守的全面落栈策略，所有临时变量都在内存(sp)中周转，没有分配寄存器**”。
*   `jal` / `call`：函数调用。
*   `a0`：一切函数的返回值都在 `a0` 里，打印 `$?` 抓的就是它。

## 3. 防翻车护身符 (自曝缺陷，防守反击)

不怕有 bug，就怕不知道 bug 在哪。主动承认以下三点，老师反而会觉得你理解很透彻。

### 🚨 缺陷一：构建时有 Bison 警告 (Shift/Reduce conflicts)
*   **现象**：`cmake --build` 的时候，终端可能会闪过 warning: shift/reduce conflicts。
*   **防翻车话术**：“是的老师，这里有冲突警告。主要来源于经典的 `if-else 悬挂问题` (dangling-else)。因为 SysY 的 else 是可选的，文法上存在二义性。为了保证项目的稳定性，我目前依赖 Bison 默认的 `优先进行 Shift (移进)` 策略解决，也就是 else 会就近匹配最近的 if。由于目前用例均能通过，就暂时没把它改写为更繁琐的无结合文法。”

### 🚨 缺陷二：生成的汇编代码行数非常多，性能很慢？
*   **防翻车话术**：“没错老师。因为时间的权衡，我的后端并没有实现图染色算法或者线性扫描的**寄存器分配 (Register Allocation)**。我采用了全部临时变量 `落栈` 的保守策略。这意味着几乎每做一次加减法，都要操作一次内存，代码膨胀率很高。但我认为对于教学编译器，首先保证 100% 的正确性、避免寄存器溢出的诡异 Bug 是第一位的。如果还有迭代时间，寄存器分配将是我第一个要优化的点。”

### 🚨 缺陷三：Float (浮点数) 处理感觉用的方案有点“怪”？
*   **防翻车话术**：“是的，这是某种程度的『逃课方案』。因为我们在中间代码 Koopa 阶段不想引入过分复杂的类型系统，所以我把所有的 float 都当作 32位的 bit pattern（比特模式）塞入 `i32` 来传输。运算时，并没有走底层的基础指令，而是通过我注入的 `__sysy_fadd` 等 helper 函数来处理。这确实不够优雅（不是 Native Float 支持），但他巧妙地在有限的时间里，保证了浮点运算能力的闭环。”
