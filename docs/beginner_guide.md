# SysY 答辩速通指南

这份文档的目标不是把源码逐行讲完，而是让第一次接触项目的人在最短时间内建立一条完整的理解链路：知道项目在做什么、每一层为什么存在、关键实现点在哪里、答辩时怎么把话讲圆。

## 1. 先记住整条流水线

这个项目本质上是一个 SysY2022 教学编译器。整体流程可以概括成：

```text
SysY 源码
  -> 词法分析
  -> 语法分析
  -> AST
  -> 语义分析 + Koopa IR 生成
  -> libkoopa raw program
  -> RISC-V 汇编
```

答辩时最重要的一句话是：前端负责把“字符串源码”变成“结构化程序”，语义阶段负责把“结构化程序”变成“可验证的中间代码”，后端负责把“中间代码”变成“目标机器代码”。

你可以把这条链理解成三次翻译：

1. 词法和语法把语言表面形式翻译成树。
2. 语义分析把树翻译成带类型、带作用域约束、带控制流的 Koopa IR。
3. 后端把 Koopa IR 翻译成 RISC-V。

## 2. 先学会讲编译原理的基本概念

答辩里经常会被问到的概念并不多，核心就是下面这些。

### 2.1 词法分析

词法分析的输入是字符流，输出是 token 流。它只关心“这一串字符是什么”，不关心语法结构。

在本项目里，词法阶段要识别：关键字、标识符、整数常量、浮点常量、字符串、运算符、分隔符和注释。对应的实现入口在当前构建链使用的前端文件中，最终由 Bison 继续做语法分析。

你可以这样解释词法分析的作用：

“它相当于把原始文本切成有意义的积木块，后面的语法分析才能按规则拼装。”

### 2.2 语法分析

语法分析的输入是 token 流，输出是语法树或抽象语法树。它解决的是“这些 token 能不能按语言规则组成一个合法程序”。

答辩里最常见的点是：

1. 表达式优先级为什么需要语法规则保证。
2. if-else 的悬挂 else 如何处理。
3. 函数定义、变量声明、数组定义这些结构怎么区分。

你可以这样讲语法分析：

“语法分析就是按照文法把 token 串还原成程序结构，保证加减乘除、if、while、函数调用这些东西都能按语言规则被正确理解。”

### 2.3 AST

AST 是抽象语法树。它比完整语法树更精炼，保留了语义分析真正需要的信息。

本项目里的 AST 定义集中在 [include/sysy/ast.hpp](include/sysy/ast.hpp)。这里有几个重点：

1. 用枚举区分节点类型，比如表达式、语句、顶层定义。
2. 每个节点只保存当前层真正需要的数据。
3. 所有节点都带源位置，方便报错。

答辩时可以这样解释为什么不用复杂的继承树：

“因为这个项目的节点类型比较固定，枚举加结构体更直接，遍历时 switch 一次就能把所有情况覆盖，读起来也比多层继承更清楚。”

### 2.4 符号表和作用域

符号表保存“名字对应什么”。作用域决定“这个名字在什么范围内有效”。

本项目语义阶段会维护栈式作用域。你可以记成三句话：

1. 进入一个块，就压入新作用域。
2. 离开一个块，就弹出作用域。
3. 查找名字时，从内到外逐层找。

这就是为什么局部变量可以遮蔽全局变量，但同一作用域里不能重复定义。

### 2.5 IR

IR 是中间表示。它比源语言更规整，比机器码更抽象，适合做统一的语义表达和后端翻译。

这个项目选择 Koopa 作为中间表示。答辩时最值得强调的点是：

“Koopa 把前端和后端解耦了。前端只要产出合法 Koopa，后端只要认识 Koopa，就不需要互相知道对方的内部细节。”

## 3. 按真实代码理解项目

### 3.1 入口在做什么

入口逻辑在 [src/main.cpp](src/main.cpp)。它做的事情很简单：

1. 解析命令行参数。
2. 调用前端把 SysY 源码解析成 AST。
3. 调用语义分析把 AST 编译成 Koopa。
4. 根据模式选择输出 Koopa 文本或 RISC-V 汇编。

这意味着主程序不是“算法重点”，真正的难点在前端、语义和后端。

### 3.2 AST 结构长什么样

AST 的核心定义在 [include/sysy/ast.hpp](include/sysy/ast.hpp)。你要先记住几个大类：

1. Program：整个编译单元。
2. TopLevel：顶层声明或函数定义。
3. Decl / VarDef：变量和常量声明。
4. FuncDef / FuncParam：函数和形参。
5. Block / Stmt / Expr：块、语句、表达式。
6. InitVal：初始化器。

答辩中如果被问“为什么 AST 要单独设计”，标准回答是：

“因为语法树太贴近文法，而 AST 需要贴近语义分析。AST 要把无关紧要的语法细节压掉，只留下后续阶段最关心的信息。”

### 3.3 语义分析到底做了什么

语义分析文件是 [src/semantic/SemanticAnalyzer.cpp](src/semantic/SemanticAnalyzer.cpp)。这部分是整个项目最核心的地方，因为它同时做了两件事：

1. 检查程序是否符合语言语义。
2. 生成 Koopa IR。

你可以把它理解成“编译器的大脑”。

这里最关键的几个职责是：

1. 收集函数签名，支持函数先声明后调用，也支持递归。
2. 维护作用域和符号表，保证名字解析正确。
3. 做类型检查，区分 int、float、void、数组、函数参数。
4. 做常量求值，用于全局初始化和数组维度。
5. 生成控制流和短路求值对应的 Koopa 结构。
6. 处理数组初始化、数组参数退化和 float 降低。

你可以把“语义分析”一句话讲成：

“语义分析不是看语句像不像，而是看语句有没有意义。它负责名字对不对、类型对不对、作用域对不对、控制流能不能成立。”

### 3.4 后端在做什么

后端文件是 [src/backend/RiscV.cpp](src/backend/RiscV.cpp)。它只消费 Koopa raw program，不再回头看 SysY 源码。

它的主要工作是：

1. 生成全局数据段。
2. 为每个函数构造栈帧。
3. 按调用约定传递参数。
4. 遍历 Koopa 指令生成对应汇编。
5. 处理 float helper 的追加。

答辩时最容易说清楚的一点是：

“这个后端没有做复杂寄存器分配，而是采用保守的全部落栈策略，所以实现清晰、调试容易、正确性更好，代价是生成代码不够精炼。”

## 4. 你必须讲明白的几个重点难点

### 4.1 为什么要先收集函数签名

因为函数可以在定义前被调用，也可以递归调用自己。先扫描一遍函数定义，把名字和类型都登记好，后面在编译函数体和表达式调用时才能正确检查参数和返回值。

### 4.2 为什么数组初始化容易出错

数组初始化不是简单“把值拍平”就行。嵌套大括号会影响元素填充位置，尤其是多维数组。正确做法是按层级展开，并在合适的位置补零。

你可以记住一个典型例子：多维数组初始化要保留行列结构，不能把叶子节点随便顺序塞进去。

### 4.3 为什么数组参数和普通数组不一样

数组作为函数参数时会退化成指针语义。也就是说，形参看起来是数组，但传入和访问方式更接近地址。这个地方如果不区分，维度就会错。

### 4.4 为什么 float 要做 helper

因为这个项目的 Koopa 主体仍然统一按 32 位值来处理。float 的运行期运算和比较，就通过内部 helper 函数来完成，再由后端生成 RV32F 相关指令。

### 4.5 为什么后端要保守落栈

因为全部临时值落栈最容易保证正确性。它避免了寄存器分配的复杂度，特别适合作业型编译器。

## 6. 三分钟讲稿模板

如果你需要一个非常短的版本，可以直接按这个顺序讲：

1. 这是什么项目：一个 SysY2022 教学编译器，把源码编译成 RISC-V 汇编。
2. 主流程是什么：词法、语法、AST、语义、Koopa、后端。
3. 核心难点是什么：作用域和类型检查、数组初始化、数组参数、float、控制流。
4. 怎么验证：正向样例和负向样例都跑通。
5. 我做出了什么：实现了从前端到后端的完整闭环。

## 7. 高频问答

### 问：为什么不用直接从语法树生成机器码？

答：因为中间表示能把前端和后端解耦。语法树更贴近语言规则，而 IR 更贴近语义和机器实现。中间加一层，可以让每个阶段的职责更清楚。

### 问：为什么语义分析和 IR 生成放在一起？

答：因为很多语义检查和 IR 生成是同一遍 AST 遍历里顺手完成的。这样实现简单，适合作业型项目，也更容易和测试样例对齐。

### 问：数组初始化为什么难？

答：因为它不是线性列表，而是有层级结构的。多维数组必须考虑嵌套大括号、补零和子数组边界，否则元素布局会错。

### 问：short-circuit 是怎么做的？

答：不是直接算布尔表达式，而是生成控制流。先算左边，满足条件才去算右边，这样能保证右边是否执行符合语言语义。

### 问：float 为什么要用 helper？

答：因为 Koopa 主体这里仍按 32 位值统一处理，float 运算由专门 helper 完成，后端再生成 RV32F 汇编配合实现。

### 问：为什么后端把值都放栈上？

答：因为这能大幅简化实现，不需要复杂寄存器分配，也更容易保证正确性。代价是代码效率不高，但对课程设计足够。

## 8. 讲解顺序建议

如果你要正式汇报，建议按这个顺序准备：

1. 项目目标和总体流程。
2. AST 和语义分析的设计。
3. 最难的三个点：数组、控制流、float。
4. 后端和栈帧策略。
5. 测试和验证结果。

如果时间很紧，就优先讲“整体流程 + 三个难点 + 测试验证”，不要试图把每个源码文件都讲成故事。答辩最重要的是逻辑闭环，不是细节堆砌。
